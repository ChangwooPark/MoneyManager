# Bashコマンドガイド

開発過程でよく使うBashコマンドを目的別に整理します。
新しいコマンドが登場するたびにこのドキュメントに追加していきます。

---

## 0. 背景知識 — Bash / ターミナル / シェルとは？

### ターミナル（Terminal）とは？

コンピューターに**テキストでコマンドを入力するウィンドウ**です。
マウスでアイコンをクリックする代わりに、キーボードでコマンドを入力してコンピューターを操作します。

```
macOS → ターミナル（Terminal）アプリまたはiTerm2
Windows → PowerShell、WSL（Windows Subsystem for Linux）
```

### シェル（Shell）とは？

ターミナルウィンドウの中で実際にコマンドを解釈・実行する**プログラム**です。
ターミナルは「ウィンドウ（画面）」、シェルは「その中で動く言語インタープリター」と考えると良いです。

```
シェルの種類:
  bash  → 最も広く使われるシェル（Bourne Again SHell）
  zsh   → macOSのデフォルトシェル（bashとほぼ同じように使える）
  sh    → 最も基本的なシェル
```

### Bashとは？

**B**ourne **A**gain **SH**ellの略です。
Linux/macOSで最も多く使われるシェル言語で、このドキュメントで説明するコマンドは全てbash（およびzsh）で動作します。

### プロンプト（Prompt）とは？

ターミナルでコマンドを入力できる状態を示す記号です。

```bash
cw-park@MacBook ~ %    # zsh プロンプト（% 記号）
cw-park@MacBook ~ $    # bash プロンプト（$ 記号）
```

`%`または`$`の後ろにカーソルが点滅していれば、コマンドを入力する準備ができている状態です。

### パス（Path）とは？

ファイルやフォルダの場所を示すアドレスです。

```
絶対パス: /Users/cw-park/project/frontend/src/components
          ↑ ルート（最上位）から始まる、常に / で始まる

相対パス: ./src/components   （現在位置を基準）
          ../components      （一つ上を基準）
```

- `/` = 最上位フォルダ（ルート）
- `~` = 現在のユーザーのホームフォルダ（`/Users/cw-park`）
- `.` = 現在のフォルダ
- `..` = 一つ上のフォルダ

### stdin / stdout / stderr とは？

プログラムがデータをやり取りする3つの経路（ストリーム）です。

| 名前 | 番号 | 説明 |
|---|---|---|
| stdin（標準入力） | 0 | プログラムにデータを入力する経路（通常はキーボード） |
| stdout（標準出力） | 1 | プログラムが結果を出力する経路（通常は画面） |
| stderr（標準エラー） | 2 | プログラムがエラーを出力する経路（通常は画面） |

`2>&1`のような記号はこのストリーム番号を使用します。

## 1. ファイル探索 — `find` / `grep` / `ls`

### 1-1. 特定ファイルを探す: `find`

```bash
find /パス -name "ファイル名パターン"
```

**用途**：ファイルがどのフォルダにあるかわからないときに検索します。

```bash
# 例：CalendarTab という名前を含むファイルを探す（node_modules 除く）
find ./frontend/src -name "CalendarTab*" -o -name "Calendar*" | grep -v node_modules
```

| 部分 | 意味 |
|---|---|
| `find <パス>` | このパス以下の全フォルダを再帰的に探索 |
| `-name "CalendarTab*"` | ファイル名が `CalendarTab` で始まるファイル（`*`は「任意の文字」） |
| `-o` | OR条件（または） |
| `\| grep -v node_modules` | 結果から `node_modules` を含む行を除去（`-v` = 除外） |

**`-not -path` で特定パスを除外：**

```bash
# 例：全PNGファイルを探して node_modules フォルダは除外
find . -name "*.png" -not -path "*/node_modules/*" | sort
```

| 部分 | 意味 |
|---|---|
| `-not -path "パターン"` | パスがこのパターンに該当する結果を除外 |
| `"*/node_modules/*"` | パスのどこかに `node_modules` フォルダが含まれるもの（`*` = 任意のパス） |

> `-o`（OR）と `-not`（NOT）は `find` の**条件演算子**です。
> `grep -v node_modules` の方式と違い、`-not -path` は探索段階でフォルダ自体を除外するためより効率的です。

---

### 1-2. ファイル内容から特定の単語を探す: `grep`

```bash
grep -n "探す単語" ファイルパス
```

**用途**：特定の変数名、関数名、設定値がどの行にあるか探す際に使用します。

```bash
# 例：MainApp.tsx で3つの単語のいずれかを含む行を探す
grep -n "CalendarTab\|refreshKey\|yearMonth" frontend/src/components/MainApp.tsx | head -30
```

| 部分 | 意味 |
|---|---|
| `grep` | ファイルからパターンを検索するコマンド |
| `-n` | 検索結果に**行番号**も表示 |
| `"単語A\|単語B\|単語C"` | 単語Aまたは単語Bまたは単語Cのいずれかを含む行（`\|` = OR） |
| `\| head -30` | 結果の上から30行のみ表示 |

```bash
# 例：CSS変数の色の値を確認
grep -n "--income\|--expense\|--accent" frontend/src/app/globals.css
```

```bash
# 例：ボタンのaria-label値を確認（E2Eテストセレクター作成前の確認用）
grep -n "aria-label" frontend/src/components/layout/MonthSelector.tsx
```

```bash
# 例：特定の定数/キー名を確認
grep -n "SESSION_KEY" frontend/src/components/AppShell.tsx
```

### 1-3. フォルダの内容リストを表示: `ls`

```bash
ls パス
ls パス | sort
```

**用途**：フォルダの中にどんなファイル/フォルダがあるか確認します。

```bash
# 例：Documents フォルダのファイルリストをソートして表示
ls ./Documents/ | sort
```

| 部分 | 意味 |
|---|---|
| `ls <パス>` | そのパスのファイル/フォルダリストを表示 |
| `\| sort` | 結果をアルファベット/番号順でソート |

よく使うオプション：

| オプション | 意味 |
|---|---|

| `ls -a` | 隠しファイル（`.`で始まる）も表示 |
| `ls -la` | 詳細 + 隠しファイルを全て表示 |

## 2. TypeScript型チェック — `npx tsc`

```bash
npx tsc --noEmit 2>&1
```

**用途**：コード作成後に型エラーがないか確認します。実際のビルドファイルは生成しません。

| 部分 | 意味 |
|---|---|
| `npx` | ローカルの `node_modules/.bin/` にインストールされたコマンドを実行するツール |
| `tsc` | TypeScriptコンパイラ（TypeScript → JavaScript変換） |
| `--noEmit` | 型チェックのみ行い、`.js` ファイルは生成しない |
| `2>&1` | エラー出力（stderr）を画面出力（stdout）に合わせて一箇所に表示 |

**結果の解釈**：
- 何も表示されない → 型エラーなし（正常）
- `error TS2345: ...` のようなメッセージ → その行に型エラーあり

## 3. 開発サーバー管理

### 3-1. 開発サーバーの起動: `npm run dev`

```bash
npm run dev > /tmp/nextdev.log 2>&1 &
echo "PID: $!"
```

**用途**：Next.js開発サーバーをバックグラウンドで起動してブラウザでアプリを確認します。

| 部分 | 意味 |
|---|---|
| `npm run dev` | `package.json` の `dev` スクリプトを実行（プロジェクトによって異なる） |
| `> /tmp/nextdev.log` | 標準出力をファイルに保存（`>` = 上書き） |
| `2>&1` | エラー出力も同じファイルに合わせて保存 |
| `&` | **バックグラウンド実行** — サーバーが起動したままターミナルを使い続けられる |
| `echo "PID: $!"` | 起動したプロセスのIDを表示（`$!` = 最後のバックグラウンドプロセスのID） |

> **PID（プロセスID）**：OSが各プロセス（実行中のプログラム）に付与する固有番号。
> 後で `kill <PID>` でそのプロセスを終了する際に使用します。

---

### 3-2. サーバーのレスポンス確認: `curl`

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000
```

**用途**：開発サーバーが完全に起動してレスポンスを返しているか確認します。

| 部分 | 意味 |
|---|---|
| `curl` | URLにHTTPリクエストを送るコマンド |
| `-s` | silentモード — 進行表示なしで静かに実行 |
| `-o /dev/null` | レスポンス本文を捨てる（`/dev/null` = 全てを吸い込む仮想ゴミ箱） |
| `-w "%{http_code}"` | レスポンスのHTTPステータスコードのみ表示 |
| `http://localhost:3000` | ローカル開発サーバーのアドレス |

**HTTPステータスコード**：

| コード | 意味 |
|---|---|
| `200` | OK — サーバー正常レスポンス |
| `404` | Not Found — そのパスが存在しない |
| `500` | Server Error — サーバーエラー |
| `000` | 接続失敗 — サーバーがまだ起動していない |

---

### 3-3. 特定ポートのプロセスを強制終了: `lsof` + `kill`

```bash
lsof -ti:3000 | xargs kill -9 2>/dev/null; echo "killed"
```

**用途**：3000番ポートを既に他のプロセスが占有していてサーバーが起動できない場合に解決します。

| 部分 | 意味 |
|---|---|
| `lsof` | List Open Files — 開いているファイル/ポートのリストを参照 |
| `-t` | PID（プロセスID）のみ表示 |
| `-i:3000` | 3000番ポートを使用中のプロセスのみフィルタリング |
| `\| xargs kill -9` | 前から受け取ったPIDを `kill -9 <PID>` コマンドの引数として渡す |
| `kill -9` | 強制終了シグナル（SIGKILL）を送信 — 無条件で即時終了 |
| `2>/dev/null` | エラーメッセージを捨てる（プロセスがなくてもエラーなしで続行） |
| `; echo "killed"` | 前のコマンドの成功/失敗に関わらず「killed」を表示 |

> **`xargs`**：前のコマンドの出力を次のコマンドの**引数（argument）**として渡すツール。
> `lsof -ti:3000` が `12140` を出力すると、`xargs kill -9` は `kill -9 12140` を実行します。

### 3-4. ログファイルの末尾を確認: `tail`

```bash
tail -5 /tmp/nextdev.log
```

**用途**：開発サーバーのログファイルの最新内容5行のみ確認します。

| 部分 | 意味 |
|---|---|
| `tail` | ファイルの末尾を表示（反対：`head` = ファイルの先頭） |
| `-5` | 最後の5行のみ表示（`-n 5` と同じ） |

> ログファイルは時間が経つほど長くなるため、最新の情報は常にファイルの末尾にあります。
> `tail` はその最新内容を素早く確認するのに役立ちます。

## 4. Playwright E2Eテスト

### 4-1. テスト実行: `npx playwright test`

```bash
npx playwright test tests/home-tab.spec.ts --reporter=list 2>&1
```

**用途**：特定のテストファイルを実行して結果をリスト形式で表示します。

| 部分 | 意味 |
|---|---|
| `npx playwright test` | Playwrightテストランナーを実行 |
| `tests/ファイル名.spec.ts` | このファイルのテストのみ実行（省略すると全テストを実行） |
| `--reporter=list` | テスト結果を一行ずつリスト形式で表示 |
| `--reporter=line` | テスト結果を一行サマリーで表示（実行中の進行状況確認に有利） |
| `2>&1` | エラー出力も画面に合わせて表示 |

**`--reporter` オプションの比較**：

| オプション | 出力形式 | いつ使うか |
|---|---|---|
| `--reporter=list` | 各テストを一行ずつ列挙 | どのテストが通過/失敗したか個別に確認 |
| `--reporter=line` | 実行中の状態を一行で更新 | テスト数が多くて実行中の進行状況をリアルタイムで見たいとき |
| `--reporter=dot` | 点（`.`）のみで表示 | 出力を最小化したいとき |

**結果の記号**：

| 記号 | 意味 |
|---|---|
| `✓` | テスト通過 |
| `✘` | テスト失敗 |
| `○` | テストスキップ |

```bash
# 全テストファイルを一括実行
npx playwright test --reporter=list 2>&1
```

## 5. ファイル削除 — `rm`

```bash
rm ファイルパス
rm ファイル1 ファイル2 ファイル3
```

**用途**：ファイルを削除します。一度に複数のファイルを列挙して削除できます。

```bash
# 例：PNGファイルを複数まとめて削除
rm calendar-tab.png stats-tab.png home-tab-phase10.png

# 例：特定フォルダ内の全PNGを削除（ワイルドカード使用）
rm .playwright-mcp/page-*.png
```

| 部分 | 意味 |
|---|---|
| `rm <ファイル>` | ファイルを永久削除（ゴミ箱に入らない、復元不可） |
| `rm ファイル1 ファイル2` | 複数ファイルを一度に削除 |
| `rm -r <フォルダ>` | フォルダとその中の内容を全て削除（`-r` = recursive、再帰） |

> ⚠️ **注意**：`rm` はゴミ箱を経由せずに即時削除します。
> 削除前に `ls` で対象を確認する習慣が重要です。

## 6. Gitコマンド — 保存とデプロイ

### Gitとは？

コードの**変更履歴を記録**するバージョン管理システムです。
いつ、誰が、何を変えたかを記録し、過去の状態に戻したり、複数人が同時に作業できるようにします。

```
ローカル（自分のPC） ─── git push ──▶ リモート（GitHub）
                    ◀── git pull ───
```

### 6-1. 変更ファイルのステージング: `git add`

```bash
git add ファイルパス1 ファイルパス2
```

**用途**：次のコミットに含めるファイルを選択（ステージング）します。

```bash
# 特定ファイルのみ追加（推奨）
git add frontend/src/components/features/calendar/CalendarTab.tsx

# 現在フォルダ以下の全変更ファイルを追加（注意が必要）
git add .
```

> `git add .` は便利ですが、`.env` のような機密ファイルが誤って含まれる可能性があるため、
> できる限りファイルを明示的に指定する方が安全です。

---

### 6-2. コミットの作成: `git commit`

```bash
git commit -m "$(cat <<'EOF'
Feat: カレンダータブの実装

- 7列月間グリッドカレンダーのレンダリング
- 日付クリック時にボトムシートを表示

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"
```

**用途**：ステージングされた変更内容をローカルリポジトリに記録します。

| 部分 | 意味 |
|---|---|
| `git commit -m "..."` | `-m` の後のテキストをコミットメッセージとして使用 |
| `$(...)` | 括弧内のコマンドを実行してその出力で置換（コマンド置換） |
| `cat <<'EOF' ... EOF` | `EOF` の間の複数行テキストをそのまま出力（heredoc文法） |

> **heredoc（`<<'EOF'`）**：複数行テキストをコマンドに渡すための文法。
> `EOF` は慣習的に使う区切り文字で、他の単語に変えても構いません。

**コミットメッセージの接頭辞の慣例**：

| 接頭辞 | 意味 |
|---|---|
| `Feat:` | 新機能追加 |
| `Fix:` | バグ修正 |
| `Docs:` | ドキュメント作成/修正 |
| `Test:` | テストコード追加/修正 |
| `Refactor:` | 機能変更なしでコード構造を改善 |

git push origin main

**用途**：ローカルのコミットをGitHub（リモートリポジトリ）にアップロードします。

| `origin` | リモートリポジトリの別名（通常はGitHubのURLを指す） |

### 6-4. 現在の状態確認: `git status` / `git log`

```bash
# 変更されたファイルリストの確認
git status

# 変更されたファイルリスト — 短縮形式
git status --short

# 最近のコミット履歴の確認
git log --oneline -10
```

| コマンド | 意味 |
|---|---|
| `git status` | 変更済み/ステージング済み/追跡なし のファイルリストを詳細説明と共に表示 |
| `git status --short` | ファイルパスのみ簡潔に表示（`M`=変更済み、`??`=追跡なし、`A`=ステージング済み） |
| `git log --oneline` | コミット履歴を一行ずつ簡略に表示 |
| `-10` | 最近10件のみ表示 |

**`git status --short` 出力例**：
```
 M src/components/MainApp.tsx     ← 変更済み（ステージングなし）
?? Documents/19-phase12-stats-tab.md  ← 新規ファイル（追跡なし）
A  frontend/tests/stats-tab.spec.ts   ← ステージング済み
```

### 6-5. GitルートパスのChek: `git rev-parse`

```bash
git rev-parse --show-toplevel
```

**用途**：現在のGitリポジトリの最上位フォルダ（ルート）のパスを確認します。現在どのフォルダにいても関係なくGitリポジトリのルートを知ることができます。

```bash
# 例：frontend フォルダの中にいてもプロジェクトのルートを知ることができる
$ pwd
/Users/cw-park/private-project/MoneyManager/frontend

$ git rev-parse --show-toplevel
/Users/cw-park/private-project
```

| 部分 | 意味 |
|---|---|
| `git rev-parse` | Gitの内部参照（コミットハッシュ、パスなど）を解析する多目的コマンド |
| `--show-toplevel` | Gitリポジトリのルートフォルダの絶対パスを表示 |

> **なぜ必要か？** `git add ファイルパス` は常にGitルートを基準とした相対パスで指定する必要があります。
> 現在位置が `frontend/` の中なら `frontend/src/...` のようにパスを合わせる必要があります。
> `git rev-parse --show-toplevel` でルートを確認すれば、パスのミスを防ぐことができます。

## 7. 特殊記号一覧

| 記号 | 名前 | 意味 |
|---|---|---|
| `\|` | パイプ（pipe） | 左の出力を右のコマンドの入力として渡す |
| `>` | リダイレクション | 出力をファイルに保存（既存内容を上書き） |
| `>>` | 追加リダイレクション | 出力をファイルの末尾に追加（既存内容を保持） |
| `2>&1` | エラー統合 | stderr（2番）をstdout（1番）に統合 |
| `2>/dev/null` | エラー捨て | エラーメッセージを /dev/null（ゴミ箱）に捨てる |
| `&` | バックグラウンド | コマンドをバックグラウンドで実行 |
| `&&` | AND連結 | 前のコマンドが**成功した場合のみ**次のコマンドを実行 |
| `;` | 順次実行 | 前のコマンドの成功/失敗に関わらず次のコマンドを実行 |
| `$!` | 変数 | 最後のバックグラウンドプロセスのPID |
| `$( )` | コマンド置換 | 括弧内のコマンドを実行した結果で置換 |
| `~` | ホームディレクトリ | 現在のユーザーのホームフォルダ（`/Users/cw-park`） |
| `.` | 現在ディレクトリ | 現在いるフォルダ |
| `..` | 親ディレクトリ | 一つ上のフォルダ |
| `*` | ワイルドカード | 「任意の文字」を意味するパターン |

## 8. 開発作業フローでコマンドが使われる順序

```
コード作成前 — 既存コードの把握
   find, grep -n, ls
   → ファイルの場所を確認、関連コードの行番号を確認
   git rev-parse --show-toplevel
   → Gitルートパスの確認（git add パスの基準点を把握）

コード作成後 — エラー確認
   npx tsc --noEmit
   → TypeScript型エラーがないか検査

ブラウザで画面確認
   npm run dev &          → 開発サーバー起動
   curl localhost:3000    → サーバーのレスポンス確認
   lsof -ti:ポート | xargs kill -9  → ポート競合時の解決

E2Eテスト実行
   npx playwright test tests/ファイル.spec.ts --reporter=line

保存とデプロイ
   git status --short     → 変更ファイルリストを簡単に確認
   git add ファイル → git commit → git push origin main
   → GitHub Actions → Vercel自動デプロイ

不要ファイルの整理
   find . -name "*.png" -not -path "*/node_modules/*"  → 対象を確認
   rm ファイル1 ファイル2 ...     → 確認後に削除
```
